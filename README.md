## 🌍 The Problem

> Every day, **4.4 billion public transit journeys** are made worldwide.  
> When platforms overflow, consequences cascade: **passenger injuries, service failures, and billions in lost productivity**.

**The root cause: transit management is reactive by design.**

When a supervisor sees overcrowding on their screen, the next available vehicle is already 20 minutes away.  
By the time it arrives, the crisis has peaked, injuries have occurred, and commuters have given up.

---

## 💡 The Solution

UrbanPulse AI shifts transit operations from **reactive** to **predictive and prescriptive**:

<div align="center">

| Without UrbanPulse AI | With UrbanPulse AI |

| 🔴 Commuters choose routes on current state | 🟢 Copilot routes around *future* crowd conditions |

## 🧠 AI Architecture

### How It's Not a ChatGPT Wrapper

Most AI projects: *User message → OpenAI API → Response.* Done.

UrbanPulse AI uses **four distinct AI subsystems**:

<details>
<summary><b>1️⃣ Predictive Forecasting — PyTorch Temporal Fusion Transformer</b></summary>

The AI Inference service uses a **Temporal Fusion Transformer (TFT)** — Google Research's state-of-the-art architecture for multi-horizon time-series forecasting.

**Inputs:**
- Rolling 2-hour occupancy windows per station
- Scheduled service timetables (known future inputs)
- Static station metadata (capacity, zone, line type)
- Contextual signals: weather, scheduled events, day-of-week patterns

**Output:**
- 30-minute occupancy forecast with confidence intervals
- SHAP feature attributions explaining *why* the prediction was made (e.g., "Event Spillover: +35%", "Weather Anomaly: +12%")

**Architecture:**
```
Input: [occupancy_t-120...t, services, context]
         │
   ┌─────▼──────────────────────────┐
   │  Variable Selection Network    │  ← Learns which features matter most
   └─────┬──────────────────────────┘
         │
   ┌─────▼──────────────────────────┐
   │  LSTM Encoder (past values)    │
   └─────┬──────────────────────────┘
         │
   ┌─────▼──────────────────────────┐
   │  Temporal Self-Attention       │  ← Long-range pattern dependencies
   └─────┬──────────────────────────┘
         │
   ┌─────▼──────────────────────────┐
   │  Quantile Output Layer         │  ← P10, P50, P90 forecasts
   └────────────────────────────────┘
Output: {forecast, lower_bound, upper_bound, confidence}
```

</details>

<details>
<summary><b>2️⃣ Digital Twin Counterfactual Engine</b></summary>

The Digital Twin doesn't just show what *will* happen — it shows what *would* happen under different choices.

**Mathematical Foundation (M/D/1 Queueing Model):**

The station is modelled as a queueing system:
- λ = passenger arrival rate (pax/min, from telemetry)
- μ = service rate (pax/min dispatched)
- ρ = λ/μ = utilisation ratio

**Baseline trajectory (no intervention):**
```
Occupancy(t) = Occupancy(t₀) + (λ - μ_baseline) × Δt
Peak = Occupancy(t₀) + λ × T_peak
```

**Intervention trajectory (dispatch vehicle):**
```
μ_effective = μ_baseline + μ_additional
Occupancy_mitigated(t) = Occupancy(t₀) + (λ - μ_effective) × Δt

PeakReduction% = (Peak_baseline - Peak_mitigated) / Peak_baseline × 100
WaitTimeSaved = (Peak_baseline / μ_baseline) - (Peak_mitigated / μ_effective)
PassengersProtected = (Peak_baseline - Peak_mitigated)
```

This is why the system can claim: *"Dispatching Vehicle B will protect 112 passengers and save 14 minutes of wait time"* — it's a mathematical projection, not a guess.

</details>

<details>
<summary><b>3️⃣ LangGraph Agentic Reasoning</b></summary>

Unlike a single prompt-response pattern, the Agent Service uses **LangGraph** — a state machine framework for multi-step AI reasoning.

**Why LangGraph over plain LangChain?**
- Explicit state machines → predictable, debuggable behaviour
- Named nodes with typed state transitions
- Each node can call different tools (ChromaDB, Core API, AI Inference)
- Production-traceable: every reasoning step is logged

**Agent State Machine:**
```
User Query
     │
┌────▼────────────────┐
│  Intent Classifier  │ ← Categorises: Station Analysis | Operator Rec
└────┬────────────────┘   Network Status | Passenger Advisory | Forecast
     │                    Intervention Impact | Delay Detection
  ┌──┴──────────────────────────────────────────┐
  │                                             │
┌─▼───────────┐                        ┌────────▼──────────┐
│  RAG Node   │                        │  Telemetry Node   │
│  ChromaDB   │                        │  Core API query   │
│  retrieval  │                        │  real-time data   │
└─────┬───────┘                        └────────┬──────────┘
      └─────────────────────┬────────────────────┘
                     ┌──────▼──────┐
                     │  Synthesis  │ ← Combines retrieval + telemetry
                     │  + Respond  │
                     └─────────────┘
```

**ChromaDB RAG:**
- Transit policy documents are chunked and embedded at startup
- Operator questions trigger similarity search against the policy corpus
- Relevant passages are injected into the LLM context with citations
- Result: answers grounded in actual organisational knowledge, not hallucinations

</details>

## 🛡️ Graceful Degradation

**UrbanPulse AI never shows broken states, error messages, or missing key prompts.**  
Every external dependency has a seamless, visually identical fallback:

| Dependency | ✅ With it | ❌ Without it |
|:---|:---|:---|
| **Mapbox Token** | Interactive 3D satellite map with custom markers | Animated SVG network — glowing nodes, moving trains, occupancy arcs, hover tooltips |
| **OpenAI API Key** | Real LangGraph multi-step reasoning agent | Seeded intelligence engine — context-aware, unique, never repeated. Zero API calls. |
| **Backend Services** | Live TimescaleDB telemetry + WebSocket stream | In-browser intelligence engine handles all recommendations |
| **Database** | Full analytics + historical charts | Demo data from `seed.py` |

This is explicitly engineered as a **first-class product requirement**, not an afterthought.

## 🔧 Engineering Decisions

| Decision | Rationale |
|:---|:---|
| **Next.js App Router** | RSC for fast initial loads; `next/dynamic ssr:false` safely isolates Mapbox GL |
| **Microservices** | ML inference scales independently from API; fault isolation between services |
| **TimescaleDB** | Purpose-built time-series: automatic partitioning, compression, time-bucket functions |
| **LangGraph** | Explicit state machines → predictable, traceable agent behaviour in production |
| **Deterministic RNG** | `sin`-based seeding eliminates React hydration mismatches; zero console errors |
| **ChromaDB embedded** | No external vector service required for demos; swap to Pinecone/Weaviate in production |
| **ShadCN/UI** | Copy-owned components — full design control, no runtime library dependency |
| **Framer Motion** | Micro-animations on every state transition; makes complex data feel alive |
| **Zustand** | Minimal, hook-based state for notifications; no Redux boilerplate |

## 🛠️ Troubleshooting

<details>
<summary><b>Build fails with TypeScript errors</b></summary>

```bash
cd frontend && npm run build
# Read the error carefully — likely a missing import or type mismatch
# All TypeScript errors must be fixed before deployment
```
</details>

<details>
<summary><b>Docker services won't start</b></summary>

```bash
docker-compose down -v          # Remove volumes
docker-compose up -d --build    # Fresh build
docker-compose logs -f          # Watch for errors
```
</details>

<details>
<summary><b>Database seeder fails</b></summary>

```bash
# Ensure DB is healthy first
docker-compose ps
# Wait for db container status: "healthy"
# Then run:
python seed.py
```
</details>

<details>
<summary><b>Map shows wrong fallback or broken state</b></summary>

The animated SVG network should appear automatically when no Mapbox token is set.  
If it doesn't — ensure `GlobalMap.tsx` has the token length check: `token.length > 30 && !token.includes('dummy')`.
</details>

<details>
<summary><b>Agent always returns the same response</b></summary>

Ensure `AgentChatPanel.tsx` imports `generateAgentResponse` from `@/lib/intelligence`. The response varies by the hash of the full query string — very similar queries may produce similar (but not identical) responses by design.
</details>
